# Camera2セッション制御 実測結果 (Phase 1)

最終更新: 2026-07-21
コード基準: `perf/camera-session-latency` ブランチ (計測コード: `db554c7`, `6bed5c3`, `f271607`)
対象実機: Sony SO-51C (ワイヤレスADB)
比較対象: `main`(`26cb267`, 計測コードなし) と本ブランチ

## 目的とスコープ

本ドキュメントは2つの異なる問いに答える。両者を混同しないこと。

- **段階A(オーバーヘッド確認)**: Phase 1で追加した計測コード自体が、録画開始レイテンシ・実測fps・スタッター率を悪化させていないか。`main`には内部メトリクスが存在しないため、比較は**外部から測れる値のみ**(MP4解析結果、録画ボタン操作からの遅延)に限定した。
- **段階B(内部ベースライン取得)**: 本ブランチの`CameraSessionMetrics`が報告する内部タイミング(T0-T9)とCaptureCallback統計を、Phase 2以降の比較基準として記録する。`main`はこの段階では無関係。

## 実施方法

- ビルドは`main`用・ブランチ用をそれぞれ1回だけ行い(`main`は`git worktree`で本チェックアウトを乱さずに構築)、以降は`adb install -r -g`での切替のみで実行した(再ビルドなし)。
- 録画の開始・停止はハードウェアカメラキー(`adb shell input keyevent 27`)で行った——`MainActivity.dispatchKeyEvent`経由の`onShutterPressed()`を使うため、画面タップのUI操作より安定。
- 解像度・露出モードの切替はスクリーンショットに基づくタップ操作で行った(座標は本セッション内で実機検証済み)。
- 各録画ごとに: 録画前の電池温度・`Thermal Status`・直近logcatの`CAM_CRITICAL`件数を記録、録画後にlogcatを全dumpしてMP4を`Movies/AuCamPRO`から回収、`tools/rec_diagnose.py`で解析。
- 温度ゲート: 電池温度が42.0℃以上の場合、45秒待機を最大8回(6分)繰り返す。条件間では明示的な冷却待ち(`<40.0℃`)を追加で挟んだ。全録画を通じて`Thermal Status`は常に`0`(NONE)、`CAM_CRITICAL`件数は常に`0`——`CAM_CRITICAL`状態には一度も入っていない。

### 実施したことの差異(仕様からの逸脱)

1. **段階Bの3条件は段階Aのブランチ実行結果を再利用した**: `1080p24 AUTO`・`1080p30 MANUAL`・`1080p60 MANUAL`はすでに段階Aでブランチ側5回分のデータがあるため、重複録画を避けて再利用した(録画自体は毎回ゼロから実施済み、後から選別しただけ)。新規に録画したのは`1080p24 MANUAL`・`1080p30 AUTO`・`1080p60 AUTO`の3条件×5回のみ。
2. **この再利用により、段階Bの6条件間でT0-T9の絶対値は完全には比較不能**: 再利用した3条件(24AUTO/30MANUAL/60MANUAL)は段階Aのmain/branch交互実行の中で発生したもので、5回中に複数回のAPKインストール切替(≒カメラの完全な再オープン)を挟んでいる。新規録画した3条件(24MANUAL/30AUTO/60AUTO)は同一インストールのまま連続録画したため、セッション/HAL状態がより「温まった」状態にある可能性が高い。実際、再利用組は`sessionConfigure`が204-483ms、新規組は206-210ms(60MANUALを除く)で、後者が明確に速い——これは**条件そのものの差ではなく、APK切替による毎回のカメラdevice再オープンコストが乗っている可能性が高い**。表はそのまま報告するが、条件間の絶対値比較にはこの限界がある。
3. **ホスト側`host_start_epoch_ns` vs デバイス側logcatタイムスタンプによる「録画開始遅延」計算は、ホスト/デバイスのクロックオフセットの影響で信頼できないと判明**——多くの値が物理的にありえない負値になった(デバイスクロックがホストより数百ms「遅れて」見える系統誤差)。この指標は採用せず、MP4自己完結型の指標(実測fps・スタッター率・最大フレーム間隔)とブランチ内部の`elapsedRealtimeNanos`ベースのT0-T9(単一デバイスのクロックのみを使うため系統誤差の影響を受けない)を主指標とした。
4. **Perfettoトレースの数値解析はしなかった**(`trace_processor_shell`をこの環境で用意していない)。T0-T9のログ行が同じ情報をテキストで提供するため、Perfettoトレース自体は「取得できること」と「アクティブキャプチャが外部指標を悪化させないこと」の確認にのみ使った(1080p30 MANUAL、ブランチのみ、5回)。

### 段階A 総合判定: **PASS**

計測コード自体が録画性能を悪化させている証拠はない。1080p60 MANUALで両方(main・branch)に共通して観測された深刻なスタッター(39-41%、後述)は、Phase 1計測コードとは無関係の既存問題である。

### 重要な副次的発見: 1080p60 MANUALは既存コードで既に不安定

`main`(計測コードなし)でも1080p60 MANUALは実測fps 20.8-55.4(median 28.72)、スタッター率28-56%という深刻な劣化を示した。**これはPhase 1が原因ではない**——mainにも同じ問題が再現する。一方、段階Bで測定した1080p60 AUTOは実測fps median 55.2、スタッター率median 8.5%と大幅に健全——`docs/VIDEO_FPS_STUTTER_INVESTIGATION_2026-07-20.md`が既に指摘した「`CONTROL_AE_MODE_OFF`(=MANUAL露出)がHAL内部フレーム損失の主要因」という結論と一致する、独立した再確認結果。ただしこの一連のテストは録画を約25秒間隔で連続実施しているため、通常の使用パターンより負荷が高い可能性があり、この劣化度がそのまま実運用の劣化度とは言えない。

### どこで時間を使っているか(ランキング)

1. **`sessionConfigure`(`createCaptureSession`)が全条件で最大の単一コスト**——正常系(24/30fps、60fps AUTO)で205-218ms、1080p60 MANUALでは483msまで悪化。これは以前の一回限りの手動計測(KDocに記載の430-1000ms)と同じ桁で、Phase 1計測でも再確認された。**Phase 2で最も投資価値が高い箇所** (Persistent Encoder Surface案がここへの根本対処)。
2. **`Event.Started`から実際の`firstVideoFrame`までのギャップが常に大きい**——`apiRecordingStart`(T7-T0、つまりアプリが「録画開始」を宣言する時点)と`firstVideo`(T8-T0)の差は正常系で200-390ms、1080p60 MANUALで426ms。アプリのイベントが発火してもまだカメラが実際にフレームを配信していない、というギャップが常に存在する。
3. **`encoderConfigure`/`encoderStart`は軽量**(正常系40-47ms/33-40ms)——MediaCodecの設定・起動自体はボトルネックではない。
4. **`abortCaptures`は通常20-30msと無視できる**が、1080p60 MANUALでは370.6msに急増——単独では説明できない異常値で、同条件の全般的な不安定さ(HAL側の負荷)を反映している可能性が高い。
5. **`setRepeatingRequest`は通常1-30msと無視できるが、1080p60 AUTOで128.2msの外れ値**——n=5と少ないため要因未確定。Phase 2で同一条件を追加測定する価値あり。

### setRepeatingRequest 呼び出し元別カウント

段階Bの各録画は「プレビュー開始→(必要なら解像度/露出設定変更)→録画開始→録画停止」という1サイクルのみなので、`SESSION_RECONFIGURE`(録画開始)と`START_REPEATING`(プレビュー開始/セッション再構築)がほぼ同数、`PARAM_UPDATE`(スライダー操作等)はこのテスト中に人間操作がないため主に**画面遷移時の自動反映**(解像度選択直後の`updateCameraParams`呼び出し等)由来、`FOCUS_REQUEST`は常に0(タップフォーカスを一度も使っていないため)。

代表例(1080p24 MANUAL, rep05、累積値): `START_REPEATING=6, SESSION_RECONFIGURE=6, PARAM_UPDATE=5, FOCUS_REQUEST=0` — この累積比からも、本テストの録画サイクルでは**録画開始由来の呼び出し(START_REPEATING+SESSION_RECONFIGURE)がPARAM_UPDATEを上回る**ことが分かる。実際のユーザー操作(スライダードラッグ)を伴うシナリオでは比率が逆転するはずで、Phase 2での「同一パラメータ省略」の効果測定にはスライダー操作を伴う別テストが必要。

## Phase 2への示唆

- 最優先: `sessionConfigure`(createCaptureSession)コスト削減 = Persistent Encoder Surface設計の検証。ここが全条件で最大のボトルネックであることが実測で確認された。
- 次点: `Event.Started`〜`firstVideoFrame`間のギャップ(200-430ms)の内訳調査——現状は「録画開始を宣言してから実際に絵が出るまで」の遅延として未分解。
- 1080p60 MANUALの深刻な不安定性(mainでも再現)は、Phase 1のスコープ外(既存の原因②)。別ブランチでのAE_ON/AE_ON+LOCK/AE_OFF比較まで持ち越す。
- `abortCaptures`除去のA/B比較、CaptureCallback再利用、同一パラメータ省略は、いずれもこのベースラインに対して前後比較すること。
