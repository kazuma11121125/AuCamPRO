## 1. アイドル再描画 ~18.6fps 残存の原因究明

### 検証結果(結論先出し)

**残存 ~18.3fps はバグではなく「本物の信号変化」による正当な配信であり、修正コードは
設計どおり動作している。** ただし Sony Video Pro を同条件で実測すると 3.5fps であり、
「正しいが、まだ Sony 水準ではない」。差の正体はメーターの**数値ラベルの表示分解能
(0.5dB)が Sony(数値ラベルなし・セグメントのみ ≈2.5dB 分解能)より 5 倍細かい**こと。
追加最適化の余地はあるが、必要性はユーザー判断(下記提案参照)。

### コミット内容の独立検証(diff 実読)

両コミットともコミットメッセージどおりの実装であることをソースで確認した:

- `7053206` P1: `app/src/main/cpp/jni/native-lib.cpp` — `g_registry`
  (`std::unordered_map<jlong, engine*>`) + `std::shared_mutex`。全 JNI 呼び出しが
  `EngineGuard`(shared_lock)でハンドル解決し、`nativeDestroy()` は unique_lock 取得後に
  erase → ロック解放後に delete。**進行中の呼び出しが返るまで delete がブロックされ、
  erase 後の呼び出しは null を見て no-op になる。UAF の窓は閉じている。**
  `NativeEngineBridge.kt` の `closed` フラグは fast-path 化(コメントも正確に更新済み)。
  `AudioEncoder.kt` は `BUFFER_EMPTY_SLEEP_MS` 2→5ms のみ(P7)。
- `7053206` P2-P4: `CameraControlViewModel.kt` — メーター 0.5dB 量子化
  (`METER_DB_STEP`)、ヒストグラム L1 距離 0.5 閾値で配信スキップ
  (`HISTOGRAM_PUBLISH_THRESHOLD`、最終**配信**値との比較なので緩慢なドリフトも最終的に
  配信される実装で正しい)。`LevelGaugeOverlay.kt` — 0.1° 量子化 + `isLevel`/色の読み取りを
  両 `Canvas` の draw ラムダ内へ移動(コンポジション位相での state 読み取りは排除済み)。
- `21bcade`: `RecordingPipeline.kt` — `dispatchAudioDeviceSetChanged()` が
  `pipelineScope.launch { sessionMutex.withLock { ... } }` で両呼び出し元
  (Settings ピッカー main スレッド / `AudioDeviceCallback` の HandlerThread)を直列化し、
  ブロッキングなネイティブ呼び出しは `withContext(Dispatchers.IO)` へ。
  `AudioDeviceRouter.candidateInputDevices()` は
  `sortedWith(compareBy({ priorityOf(it.type) }, { it.id }))`(53-58行)。

### 実機再計測

| 測定 | 値 |
|---|---|
| AuCamPRO アイドル再描画 (gfxinfo, 30秒, 2回) | **549 frames / 30s = 18.3fps**(2回とも 549 — 再現性高い) |
| AuCamPRO CPU (`measure2.sh` 40秒) | **29% / 1コア**(main 323 ticks, RenderThread 305 ticks = 7.6%) |
| Sony Video Pro アイドル再描画(同室・同時刻・メーター/水準器表示あり) | **105 frames / 30s = 3.5fps** |
| Sony Video Pro CPU (同 40秒) | **20% / 1コア**(RenderThread 38 ticks = **0.95%**) |

修正前 31fps → 18.3fps は確認。CPU 37%→29% も Sonnet の計測と一致。

### 何がまだ描画しているのか — screencap 差分での画素レベル特定

1秒間隔スクリーンショット 16 枚の連続ペア差分(領域別、閾値付き):

| 領域 | 変化したペア数 |
|---|---|
| 音声メーター (左上, x[42,229] y[232,409]) | **15/15 (毎回)** |
| 水準器 (中央, x[729,1043] y[400,587]) | 3/15 |
| ヒストグラム (左下) | **0/15 (完全静止)** |
| 右コントロールパネル | **0/15 (完全静止)** |

同じ手法で Sony Video Pro のメーター領域は **2/11** しか変化しない。

**メーターがほぼ唯一の駆動源。** メーター表示は実測 -31〜-35dB 前後(「静かな部屋」でも
PC ファン等の暗騒音がこのレベルで常時存在し、有線ヘッドセットマイク経由)。ピーク/RMS は
50ms ごとに 0.5dB 量子化ステップを普通に跨ぐため、20Hz ポーリングのほぼ毎ティックで
StateFlow が配信される → 18.3fps ≈ 20Hz × (1 − 等値縮退分)。数値が綺麗に説明できる。

### 他の常時無効化源の監査(全て白)

- `AudioStatsRow` (MainScreen.kt:1373): `meterState` を collect するが AUDIO タブ内のみで
  アイドル時(CAMERA タブ)は非コンポーズ。右パネル画素も 0/15 で静止確認。
- REC インジケーター (MainScreen.kt:686-699): `rememberInfiniteTransition` は
  `state.isRecording` 分岐の内側でのみ生成 — 非録画時に VSync 購読なし。ゲート確認済み。
- `FocusReticleOverlay`: アニメーション/フレームループなし。タップフォーカス時のみ描画。
- `ThermalMonitor`: ポーリングなし、`OnThermalStatusChangedListener` のイベント駆動のみ。
- `DeviceOrientationTracker`: 素の `var` に 90° 量子化値を保持するだけ。Compose state 非接続。
- AWB/AF パッシブ測定 (RecordingPipeline.kt:594-627 の `frameCount % 10` パス):
  ViewModel 側 (CameraControlViewModel.kt:118-150) に 50K / 0.1 diopter のデッドバンドあり。
  右パネル静止の画素証拠とも整合。

### 判定と次の一手の提案

**判定: 「正当な配信」であり P2-P4 の実装ミスではない。ただし Sony 比でまだ ~15fps /
RenderThread ~6.7pt / CPU ~9pt の差があり、詰める価値はある(任意)。**

差の構造: バー本体は 24 セグメント × 60dB レンジ = **2.5dB/セグメント**
(AudioMeterBar.kt:239-241)なので、0.5dB 刻みの配信のうち約 8 割はバーの画素を 1px も
動かさず、**数値 dBFS ラベル(0.1dB 表示精度)だけを書き換えている**。Sony は数値ラベル
自体を持たない。

提案(P8 候補、実装は別途判断):
1. **配信を 2 系統に分離**: バー用はセグメント境界 (2.5dB) を跨いだときのみ配信、
   数値ラベル用は 4-5Hz に間引いて 0.5dB 量子化を維持。実装は
   `CameraControlViewModel.startMeterPolling()` 内の publish 条件変更のみで、
   見た目のバー応答性(20Hz)は変化時には保たれる。期待値: アイドル 18.3fps → 2-4fps、
   RenderThread ~7.6% → ~1-2%、CPU 29% → 24-25% 程度(Sony 20% にほぼ並ぶ)。
2. 代替案(より小さく): 非録画時のみ `METER_POLL_INTERVAL_MS` を 50→100ms に。
   半減止まり (~10fps) なので効果は限定的。案 1 を推奨。
3. 水準器 (3/15) とヒストグラム (0/15) は追加対応不要。

## 3. `deviceId` 副キーに関するレビューコメントの監査

> deviceId は再接続や再起動で変わり得るため永続識別子として不適。将来の同一物理デバイス
> 復帰・ユーザー設定保持で課題。productName/type/チャンネル数等での照合や選択デバイスの
> 永続化を提案。

**判定: 現状コードの潜在バグの指摘ではなく、スコープ外と既に整理済みの将来機能の話。
今回のコミットに対するアクションは不要。**

根拠(コード実読):

- `deviceId` の用途は `candidateInputDevices()` (AudioDeviceRouter.kt:58) の
  **同一優先度層内のソート副キーのみ**。戻り値は
  `onAudioDeviceSetChangedLocked()` (RecordingPipeline.kt) が**その場で**フォールバック
  順走に消費する。デバイス増減のたびに `getDevices()` を再列挙して作り直すため、
  再接続・再起動をまたいで id が参照される経路は存在しない。
  `RecordingPipeline.activeInputDeviceId` も現行セッションの表示/照合用で、
  ホットスワップ時に毎回上書きされる。
- 永続化されるのは `UserPreferencesStore.kt:63-65` の `InputKind` **enum 名のみ**
  (Auto/Usb/Wired/BuiltIn の「種別」)。deviceId はどこにも保存されない。UI も実機確認
  どおり種別ピッカー(自動/USB Audio/有線ヘッドセット/内蔵マイク)であり、
  「この特定の USB インターフェース」を選ぶ UI 面は存在しない。
- したがって「deviceId は永続識別子として不適」は Android の事実として正しい
  (`AudioDeviceInfo.getId()` は接続中のみ有効な一時ハンドル)が、**本コードは deviceId を
  永続識別子として使っていない**。コミットの主張(「今この瞬間に複数 USB が刺さっている
  ときの選択を決定的にする」)の範囲では正しく、範囲外の性質を要求するのはレビューの
  射程超過。
- 前セッションのスコープ整理(per-device sticky 選択 = `productName`+`type` 永続化は
  別機能として見送り)とも一致。なお参考: 仮に将来その機能を作る場合も、同一モデルの
  USB IF を 2 台挿すと `productName` は衝突するため、**id 副キーはその場の決定性担保
  として残す価値があり、置き換えではなく「永続照合を上に重ねる」形が正しい**。
- 推奨: レビューコメントは「将来 per-device 選択機能を作るときの設計メモ」として
  バックログに留める。コード変更・コメント追記とも不要(AudioDeviceRouter.kt:55-57 の
  既存コメントが主張範囲を正しく限定している)。

## 追加ラウンド: P8実装後の再検証 (2026-07-17 深夜 ラウンド2)

実施: Fable (読み取り専用の監査・計測セッション。コード変更なし)
対象コミット: `3b8f3e0` (P8: バー配信/数値ラベル配信の分離)
実機: SO-51C (QV7713DGCD)。インストール済みビルド 23:32:52 = コミット(23:34:50)直前のビルドで、
P8 コードを含むことを再描画レートの実測 (下記) でも裏取り済み。

### 1. 「11.5fps vs 予測 2-4fps」ギャップの解明

**結論: P8 の実装は正しい(ゲートのマッピングは AudioMeterBar と完全一致)。ギャップは
(a) ラベル 5Hz 配信の設計上のフロア ~5fps + (b) セグメント境界フラッピング ~6/秒 の和で
完全に説明でき、フラッピング仮説は実測で確認された。**

#### 実装検証 (diff 実読)

- `meterSegmentIndex()` の `(db+60)/2.5` は AudioMeterBar.kt の
  `dbToFraction(db)*SEGMENT_COUNT` = `((db+60)/60)*24` と数学的に同一 (60/2.5=24)。
  境界集合も `litSegments`/`peakSegment` の切り捨てと一致しており、**ゲートと描画のドリフトなし**。
- 端点の微差 2 つはいずれもゲート用途では無害と確認:
  ちょうど 0dBFS で gate は seg 24 / 描画側は 23 に coerce (余分な状態が 1 つ増えるだけ)、
  -60dBFS 以下の「···」表示への切替は labelDue 経路 (最大 200ms 遅延、知覚不能)。
- `clippingChanged` は OR 条件に入っており即時配信。ロジック上スロットルされ得ない (§3)。

#### 実測: 再描画レートの再現とフラッピングの直接確認

- gfxinfo 30秒×2回 (リセット後): **350 / 315 フレーム = 11.7 / 10.5fps** — Sonnet の 11.5fps を再現。
- メーター領域スクリーンショット 12 枚 (0.6秒間隔) のピークラベル実読:
  **-30.0, -32.0, -35.0, -33.0, -31.5, -33.0, -33.0, -33.5, -33.0, -33.5, -32.5, -34.0**
  (L/R 同値 — ヘッドセットマイクはモノラルで両chに複製されており、ゲート 4 系統は実質 2 系統)。
- セグメント境界は -60 から 2.5dB 刻み = **…-35.0, -32.5, -30.0…**。暗騒音ピーク (平均 ≈-32.8,
  振れ幅 -30.0〜-35.0) は **-32.5 境界のド真ん中に座っている**。しかも境界 (2.5 の倍数) は
  量子化ステップ (0.5) のちょうど倍数なので、**最小の量子化 1 ステップ (-33.0↔-32.5) がそのまま
  セグメント跨ぎになる — デッドゾーンがゼロ**。仮説どおりのフラッピングが常時発生している。

#### fps の分解 (数値が閉じる)

| 成分 | 寄与 |
|---|---|
| labelDue (200ms 周期、0.5dB 量子化値は 200ms 間隔ではほぼ毎回変化 → StateFlow 等値縮退が効かない) | **~5fps (設計上のフロア)** |
| segmentChanged (フラッピング: 20Hz ポーリングの ~30% のティックで境界跨ぎ) | **~5.5-6.5fps** |
| 合計 | **~11 fps ≈ 実測 10.5-11.7fps** |

つまり**ラウンド1 の「2-4fps」予測自体が過大楽観だった**: 自ら提案した「ラベル 4-5Hz」は
生きた部屋では単独で ~5fps を保証するので、フラッピングが皆無でも 2-4fps には到達し得ない。
予測誤差 = ラベルフロアの見落とし (+5fps) + フラッピングの見落とし (+6fps)。実装ミスではない。

#### 判定と提案 (P9 候補)

**ヒステリシスは実装する価値あり (小さく・低リスク・実測された病理に直撃)。** 暗騒音が境界を
跨ぐのはエッジケースではない: ノイズ振れ幅 ±1〜1.5dB に対しセグメント幅 2.5dB なので、
定常信号が境界を跨ぐ確率は常に高い (今回はピークがほぼ境界中心)。

- **P9-a (推奨)**: セグメント跨ぎ判定に ±0.5dB (= METER_DB_STEP 1 個分) のシュミットトリガー。
  lastPublishedSeg から上がるには「上側境界 +0.5dB 以上」、下がるには「下側境界 −0.5dB 未満」を
  要求。-33.0↔-32.5 の往復では発火せず、-32.0 到達で上昇 / -34.0 で下降。バー表示の遅れは
  最悪 1dB 未満 (セグメント分解能 2.5dB の半分以下) で知覚不能。
  期待値: segment 配信 ~6/s → <1/s、**合計 ~11.5fps → ラベルフロアの ~5fps**、CPU −1〜2pt。
- **P9-b (任意、ユーザーの好み次第)**: `LABEL_PUBLISH_INTERVAL_MS` 200→333〜500ms (3〜2Hz)。
  P9-a と併せて **~2-4fps、CPU ~21-22% で Sony (3.5fps/20%) とほぼ並ぶ**。数値ラベルの
  更新体感が 5Hz→2-3Hz に落ちるのはトレードオフ (Sony には数値ラベル自体がないことに留意)。
- P9 を入れない場合でも 11.5fps/23-24% は「数値ラベルという Sony にない機能の対価」として
  停止可能な水準ではある。ただし P9-a は数行で済みリスクも低いため、やらない理由が薄い。

### 2. Janky 1.5% → 5.23% の検証

**結論: 実回帰ではない。分母の縮小による算術的な見かけ上の上昇で、絶対ジャンク数はむしろ横ばい〜微減。**

- 再計測 (30秒リセット窓 ×2): **20/350 = 5.71%、21/315 = 6.67%** — 5.23% は一発ノイズではなく再現する。
- **絶対数で比較**: 今回 ~20-21 枚/30秒 (0.67-0.70/s)。比較対象は
  P1-only 期の 1.5% (≈31fps × 30s ≈ 930 フレーム → **~14 枚/30秒**)、
  修正前計測の 2.6% (935 フレーム → **~24 枚/30秒**)。つまり絶対ジャンク発生率は従来と同じ帯域。
  変わったのは分母だけで、P8 が意図どおり総フレーム数を 62% 削ったため率が跳ね上がって見える。
- ゲートロジック起因の「バースト重畳」は構造上あり得ない: 3 条件は OR で **1 ティックにつき
  publish は最大 1 回、条件が同時成立しても `copy()` の内容・重さは同一**。バッチングは存在しない。
- 副次要因として、配信間隔が 200ms 以上空くと CPU/GPU クロックがアイドルに落ち、次フレームの
  レイテンシ裾が太る効果はある (99th %ile 23ms / 50th 9ms — 軽微)。アイドル画面には連続
  アニメーションが存在しないため、孤立したメーター更新 1 枚が deadline を外しても体感上の
  「カクつき」にはならない。
- **対応不要。** なお P9 実装後は分母がさらに減り率は 10% 超に見えるはずなので、今後この指標は
  **「janky %」ではなく「janky 枚数/30秒」の絶対値で追跡すること**を推奨する。

### 3. 回帰スイープ (P8 差分に限定)

- **メーターの生存・追従**: 計 19 枚のスクリーンショットで、ラベル・ピークセグメント・RMS バーが
  連続サンプル毎に暗騒音 (-30〜-35dB) を追従して変化。固着・凍結なし。境界跨ぎ→即時配信は
  §1 の実測 (~6 回/秒) そのものが動作証明になっている。
- **クリッピング即時性 (コード検証)**: `clippingChanged` は OR ゲートの独立項で、点灯
  (peak > -0.1dBFS → 同一ティックで held=true → 即 publish) と 3 秒ホールド後の消灯の両方が
  labelDue の間引きを一切経由しない。**スロットルされる経路は存在しない。**
  ※物理的なクリッピング誘発は今回も不可 (システム音ミュート + 出力が有線ヘッドセット直行で、
  スピーカービープ・バイブレーション最大振幅ともヘッドセットマイクには暗騒音以下)。

## 追加ラウンド3: P9-a実装後の再検証、Sony比較の最終確認 (2026-07-18)

対象コミット: `7f673bd`(P9-a: `meterSegmentIndexHysteresis()` の追加と `startMeterPolling()`
のセグメント判定呼び出しの差し替え。差分は `CameraControlViewModel.kt` の 37 行のみ)。
実機: Xperia SO-51C(QV7713DGCD)。Opus による検証(前ラウンドまでは Fable/Sonnet)。

### 結論先出し(3点)

1. **P9-a の実装は正しい。** シュミットトリガーの境界計算・全エッジケース(初回 tick /
   多セグメント跳躍 / 境界内振動)ともにバグなし。**検証合格。**
2. **6.5fps/19% は再現しなかった。** 本日の実測は **アイドル ~15.5fps(460-483 frames/30s)/
   CPU 27%**。原因はアイドル再描画レートが**環境依存(部屋の騒音レベル)**であること。
   Sonnet の 6.5fps/19% は「静音・静止シーン」の好条件スナップショットであり、
   安定した再現可能ベンチマークではない。「5fps フロアを下回った」という前提自体が誤り
   (6.5 > 5。かつラベル 5Hz フロアは StateFlow 等値縮退で*ソフト*なので、真の静音下でのみ
   5fps を下回り得るが、今回はそれではない)。
3. **「Sony を下回る(19% < 20%)」は成立しない。** 同室・背中合わせ実測で
   **AuCamPRO 15.5fps/27% vs Sony 6fps/20%**。19% vs 20% は*非同一条件*(AuCamPRO=静音時 /
   Sony=数時間前)の比較であり、計測アーティファクト。タスクが警戒した通りのケース。

### 1. ヒステリシス実装の正当性(diff 実読 + エッジケース解析)

```kotlin
private fun meterSegmentIndexHysteresis(db: Float, lastPublishedSeg: Int): Int {
    val rawSeg = meterSegmentIndex(db)
    if (rawSeg == lastPublishedSeg || lastPublishedSeg == Int.MIN_VALUE) return rawSeg
    if (kotlin.math.abs(rawSeg - lastPublishedSeg) > 1) return rawSeg
    val boundaryDb = METER_SEGMENT_DB_FLOOR + METER_SEGMENT_WIDTH_DB * maxOf(rawSeg, lastPublishedSeg)
    return if (rawSeg > lastPublishedSeg) {
        if (db >= boundaryDb + METER_DB_STEP) rawSeg else lastPublishedSeg
    } else {
        if (db < boundaryDb - METER_DB_STEP) rawSeg else lastPublishedSeg
    }
}
```

- **境界計算が両方向で正しい。** `meterSegmentIndex` はセグメント N を dB 区間
  `[FLOOR+N·WIDTH, FLOOR+(N+1)·WIDTH)` に割り当てる。`maxOf(rawSeg, lastPublishedSeg)` は
  隣接 2 セグメント間の境界をちょうど選ぶ ── 上昇時は現セグメント上端 `FLOOR+WIDTH·(L+1)`、
  下降時は現セグメント下端 `FLOOR+WIDTH·L`。上昇は `db ≥ 境界+0.5dB`、下降は `db < 境界-0.5dB` を
  要求する正しいシュミットトリガー。全値が 0.5dB の倍数なのでマージンは満たし得る。
- **初回 tick(`lastPublishedSeg == Int.MIN_VALUE`): 即 rawSeg 採用。** バグなし。
- **多セグメント跳躍(手拍子等、`|Δ|>1`): 即 rawSeg 採用**(境界計算前に return)。
  ヒステリシスは単一セグメント揺れ用であり、明白な実レベル変化には遅延を課さない。バグなし。
- **境界内振動(±0.5dB マージン内で永久に振動): 現セグメントを保持。** これは*仕様通り*で
  バグではない。陳腐化は最大 1 セグメント(2.5dB 幅)に限定され、レベルが決定的に動いた瞬間に
  解消する。暗騒音のフラッピング抑制がまさに目的。
- **判定: 実装合格。コード欠陥なし。** 後述の体感回帰確認とも整合。

### 2. アイドル再描画/CPU の再計測 ── 6.5fps/19% は再現せず、環境依存が真因

| 計測(同室・同時刻・アイドルプレビュー STANDBY) | 実測 |
|---|---|
| AuCamPRO P9-a 再描画 (gfxinfo 30s ×4) | **483 / 461 / 471 / 460 frames = 15.3-16.1fps** |
| AuCamPRO P9-a CPU (measure2.sh 30s) | **27% / 1コア**(main 207 ticks=6.7%, RenderThread 183=5.9%) |
| AuCamPRO P9-a Janky (絶対数) | **15-29 枚/30s(3.2-6.3%)** ── 95%tile 14-15ms、大半が締切内 |

Sonnet の 6.5fps(195 frames)/19% は再現できず、~15.5fps/27% に上振れた。**これはバグでも
P9-a の失敗でもなく、アイドル再描画レートが本質的に環境依存であることの現れ。**

- **主駆動源は従来通りオーディオメーター**(P1〜ラウンド2 で一貫して特定済み)。今回の環境は
  暗騒音の実信号があり(スクショで L=-39.0dB / R=-51.5dB、静音フロアではない)、ラベル 5Hz 配信
  + セグメント変化が活発。
- **騒音が P9-a を部分的に無力化している。** レベルが tick 間で 1 セグメント幅(2.5dB)を超えて
  振れると `|Δ|>1` バイパスが発火し、ヒステリシスを飛ばして毎 tick 配信し得る。よって
  **P9-a の再描画削減効果は「静音時に最大、騒音時に小」**。これは設計上正しい(大信号は即時反映
  すべき)が、賑やかな部屋では 6.5fps に届かない理由そのもの。
- 「5fps フロアを下回った」前提は二重に誤り: (a) 6.5 > 5 で下回っていない。(b) ラベル 5Hz は
  *ハードフロアではない* ── `labelDue` 配信でも量子化値が前回と等しければ `MutableStateFlow` が
  等値縮退し再描画は起きない。真の静音下では 5fps 未満に落ち得るが、今回は逆に活発化していた。
- **副次的な再描画源としてヒストグラム表示**(左下、L1距離 >0.5 で ~5Hz gate、Sony にはない機能)が
  存在するが、今回の静止シーンでは L1 ゲートがほぼトリップせず**寄与は分離できていない・軽微と推定**
  (HistogramReader スレッド 38 ticks はサンプリングコストであって再描画コストではない)。主因は
  あくまでオーディオメーター。
- **P9-a の実効果の独立 A/B(親コミット P8 のビルド比較)は本ラウンドの read-only 制約により未実施**
  (ソース改変に当たるため実行不可)。ただしラウンド2 の静音下実測 P8=11.5fps と Sonnet の
  P9-a=6.5fps が同条件なら、その差 ~5fps は P9-a が消したセグメントフラッピング分と整合的で、
  **静音下では P9-a は効くと解析的に支持される。**

### 3. Sony との最終比較 ── 「下回る」は成立せず(計測アーティファクト確定)

同室・背中合わせで両アプリを実測:

| アプリ(アイドル STANDBY 同室) | 再描画 (gfxinfo 30s) | CPU (30s) |
|---|---|---|
| AuCamPRO P9-a | **460-483 frames = ~15.5fps** | **27%**(main 207 + RT 183) |
| Sony Video Pro | **172 / 194 frames = ~6fps** | **20%**(main 104 + RT 79) |

- **Sony 自身の再描画がラウンド1 の 3.5fps → 今 ~6fps に上昇。** 同一バイナリが上振れした事実が、
  「部屋が当時より賑やかになった」ことを直接証明する ── これが 19-vs-20 比較を無効にしたアーティ
  ファクトの正体。タスクが疑った thermal/環境差はこれで裏取り完了。
- **同一条件では AuCamPRO は Sony のアイドル効率に及ばない**(27%/15.5fps vs 20%/6fps)。
  差は UI 由来(AuCam の main+RT = 12.6% vs Sony ~5.8%)。
- ただし差の大半は **Sony が持たない常時ライブ要素のコスト**: 数値 dBFS ラベル(5Hz)+ ライブ

  不可能、静音時に収束し得るのみ。** 「19% で Sony 越え」は達成マイルストーンではなかった。

### 4. 回帰スイープ(P9-a 差分に限定 ── セグメント判定関数のみ)

  `clippingChanged` が即時配信を強制する経路は無改変(コードで確認)。
- **クラッシュ: コミット後の AuCamPRO クラッシュ/ANR ゼロ。** dropbox 直近 ANR は
  com.twitter.android(07-18 10:42)と com.microsoft.skydrive(07-17 13:31)で無関係。
  AuCamPRO の native crash は全て 07-16(コミット前、既知の GAIN 調査分)。logcat に FATAL/
  SIGSEGV/ANR なし。
